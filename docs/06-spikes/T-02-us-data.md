# T-02 スパイク結果: 米国株の無料データ取得(yfinance)

最終更新: 2026-07-10 / 実施: Sonnetサブエージェント(T-02担当)

## 結論(先頭)

- **S&P500全銘柄(503銘柄)の日次一括取得は yfinance のレート制限内で安定して成立する。**
- ファンダメンタルズ(株価・PER・PBR・時価総額)一括取得: **503/503銘柄が成功(成功率100%)**、所要時間 **153.1秒(約2分33秒)**、1銘柄あたり平均 **0.30秒**。
- モメンタム用の13ヶ月分日次終値の一括取得(`yf.download`バルク): **503/503銘柄が成功(成功率100%)**、所要時間 **27.7秒**。
- ただし**セッション開始直後の初回リクエストで一時的に全滅する現象を実際に観測した**(下記「観測された問題」参照)。これはレート制限というより crumb/cookie 認証の初期化失敗と考えられ、**数十秒待って再試行すれば自然回復する**。日次バッチではこの現象への耐性(起動直後の1回失敗を許容するリトライ)が必須。
- S&P500構成銘柄リストは Wikipedia の "List of S&P 500 companies" ページを `pandas.read_html` でパースする方法で再現可能に取得できた(503銘柄、キャッシュ機構あり)。

## 1. S&P500構成銘柄リストの取得方法

- **採用方法**: `https://en.wikipedia.org/wiki/List_of_S%26P_500_companies` を `requests`(明示的UAヘッダ付き)で取得 → `pandas.read_html` で1つ目のテーブルをパース → `Symbol`列を抽出。
  - pandas.read_html にURLを直接渡すとUA拒否されるケースがあるため、requestsで先に取得したHTML文字列を渡す方式にした(再現性を優先)。
  - yfinanceのティッカー表記との差異吸収: `BRK.B` 等の `.` を `-` に置換(`BRK-B`)。
- **取得結果**: 503銘柄取得(2026-07-10時点)。
- **キャッシュ**: `pipeline/spikes/out/sp500_tickers.csv` に保存し、以降はネットワーク不要で再実行可能。
- **再現性の懸念**: Wikipediaのテーブル構造変更に弱い(壊れやすい取得元)。本番パイプライン(T-03以降)では、変更検知のための行数チェック(例: 495〜510件の範囲外なら異常とみなす)を組み込むことを推奨。

## 2. ファンダメンタルズ(株価/PER/PBR/時価総額)一括取得の実測

### 手法
`yfinance.Ticker(ticker).info` を1銘柄ずつ順次呼び出し、`currentPrice`(なければ`regularMarketPrice`)、`trailingPE`、`priceToBook`、`marketCap` を取得。失敗時は指数バックオフ(2秒→4秒→8秒)で最大3回リトライ。

### 観測された問題(重要)

1回目の実行(`--limit 15`)で**503銘柄取得を試みる前の最初のパイロットが100%失敗**した。全銘柄が `429 Too Many Requests` → `JSONDecodeError: Expecting value: line 1 column 1 (char 0)` で3回リトライしても失敗。所要時間は112.0秒(7.46秒/銘柄、リトライ待機込み)。

その直後、単発の `yf.Ticker('AAPL').info` は問題なく成功。10秒ほど間を置いた**再実行(同じ15銘柄)は15/15成功(0.32秒/銘柄)**。以降の100銘柄・503銘柄の実行はすべて成功率100%だった。

**解釈(観測に基づく推測)**: yfinanceはYahoo! Financeのcrumb(CSRFトークン)をセッション開始時に取得する。このcrumb取得自体が失敗すると、後続の全リクエストが429/空レスポンスで連鎖的に失敗する。これは「大量リクエストによるレート制限」ではなく「セッション初期化の一時的な失敗」であり、**数十秒の待機を挟んだ再試行で解消する**ことを実際に確認した。原因の断定はしていない(Yahoo側の状態は不明)が、現象と再現手順は上記の通り実測済み。

## 3. モメンタム用の13ヶ月分日次終値の一括取得(実測)

### 手法
`yf.download(tickers, period="13mo", interval="1d", group_by="ticker", threads=True)` で503銘柄を1回のバルク呼び出しで取得。

### 結果

| 項目 | 値 |
|---|---|
| リクエスト銘柄数 | 503 |
| 取得成功銘柄数 | 503 (**100.0%**) |
| 取得失敗銘柄 | なし |
| 所要時間 | **27.7秒** |

ファンダメンタルズ(1銘柄ずつ順次、153秒)と比べて、株価履歴は`yf.download`のバルク・マルチスレッド取得により1桁近く高速(27.7秒)。モメンタム計算(12-1ヶ月リターン)用のデータ取得はボトルネックにならないことを確認。

## 4. リトライ戦略の推奨(T-05実装への提言)

1. **セッション開始直後の全滅パターンへの耐性**: 日次バッチの最初のリクエストが失敗した場合、単純リトライではなく「30〜60秒待機してから再試行」を1回挟む設計にする(本スパイクの指数バックオフ2→4→8秒では、たまたま3回目までに収まらなければ全滅しうる場面を実測で観測している)。
2. **個別銘柄の一時的失敗**: 指数バックオフ(2秒→4秒→8秒、最大3回)で十分。本測定では503銘柄すべてが1回目で成功したため「個別銘柄が孤立して失敗するケース」の実測データは今回得られていない(**不明点として明記**)。将来の本番運用では失敗ログを継続的に記録し、傾向を把握することを推奨。
3. **品質ゲート**: T-05の設計に、取得成功率が閾値(例: 95%)を下回った場合は前日データにフォールバックする仕組みを入れる(05-work-breakdown.mdのT-05要件と整合)。
4. **所要時間の余裕**: フル実行でファンダメンタルズ153秒+株価履歴28秒 ≈ **3分程度**。GitHub Actionsの無料枠(1回あたりの実行時間制約)には十分収まる。ただしセッション初期化失敗のリトライ待機(30〜60秒)を見込んでも合計5分以内に収まる見通し。

## 5. 未確認・残課題

- crumb/cookie初期化失敗の**再現条件は特定できていない**(何度も連続実行すると発生しやすいのか、時間帯依存か等は未検証)。T-05の本番実装時に複数日にわたって観測を蓄積し、発生頻度を確認することを推奨。
- データ品質の外部サイトとの厳密突合はT-04で実施予定(本スパイクでは値のオーダー確認のみ)。
- PER/PBR欠損銘柄(REIT、金融特殊業態、直近赤字企業等)の扱いはT-04の指標計算モジュールで定義が必要。
